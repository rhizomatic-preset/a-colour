# Phase distillation — build-time-distilled common-noun lookup

Date: 2026-05-17
Library: small (xkcd ∪ CSS, ~980 entries)
Engine: literal (TF-IDF) + handcurated + static expanders + distillation
Branch: `feat/colour-distillation` (worktree)

This is the per-category lift table for the build-time-distilled common-noun
lookup (Phase distillation). The lookup is a static JSON file
(`src/generated/colour-distillation.json`, ~96 KB, 693 entries) authored by
in-session Claude reasoning over physical-world common nouns. Each entry is
`{ family, hex, confidence, alternates? }`. At runtime, single-token queries
that hit the lookup short-circuit the TF-IDF + expander stack: the runtime
filters library entries to those whose `getPrimaryColorName` matches the
distilled `family`, ranks them by Oklab distance to the distilled hex, and
returns the top-N. Alternates only fire when the primary family has fewer
than N candidates.

## Why this approach works

The previous Phase 2B attempts failed for two reasons:

1. **Mean-GloVe-of-tokens has no semantic bridge** from non-colour nouns
   (`wood`, `kirby`, `iron`) to colour vocabulary. Cosine in the GloVe space
   is near-random for these — see `feat/phase-2b-rerank`'s commit message.
2. **The strict-fallback hybrid never reached the engine** because TF-IDF
   always produced *some* score above threshold once the expander layer had
   fired. Even with a perfect engine, the gate was wrong — see
   `phase-2b-glove-300d.md`'s threshold sweep.

The distillation lookup sidesteps both: there's no embedding, no cosine, no
gate. Each entry is a hand-reasoned `(word → family, hex)` mapping. The
runtime is a 20-line family filter + Oklab nearest-neighbour. For tokens not
in the lookup, the existing TF-IDF + expander stack runs unchanged.

## Implementation notes

- **Lookup file**: `src/generated/colour-distillation.json`. 693 entries,
  ~96 KB on disk, ~120 KB at parse time. Negligible against the ~1 MB PWA
  precache budget.
- **Confidence levels**: `high` (480 entries — widely-recognised single
  association), `medium` (170 entries — primary family agreed-on but
  alternate plausible), `low` (43 entries — contested/context-dependent).
  At runtime the lookup is skipped for `confidence: low` entries so they
  defer to TF-IDF.
- **Token vs library colour-name conflicts**: when a token IS an exact xkcd
  / CSS colour name (e.g., `coral`, `mustard`, `salmon`), the lookup is
  *not* given an entry for that token — the literal-name path keeps
  preference. The verifier script (`scripts/verify-distillation.ts`) walks
  the eval set after every lookup edit and flags any case where the lookup
  would intercept-and-disagree with the case's expected family or name.
- **Family-filtered Oklab ranking**: for a distilled
  `{ family: "orange", hex: "#CB6D17" }`, the runtime first restricts the
  library to entries where `getPrimaryColorName(entry.hex) === "orange"`,
  then ranks within. This is critical because Oklab's nearest neighbour to
  `#CB6D17` in the full xkcd ∪ CSS library is `Browny Orange` (which the
  band classifier calls `brown`, not `orange`). Filtering first preserves
  the family intent.
- **Alternates** are tried only when the primary family pool can't fill
  topN — they're not blended with the primary. So `wood → brown` is always
  three brown entries; alternates only show up for very-thin families like
  `silver` (4 entries) and `black` (1 entry) where the primary band can't
  serve topN.

## What this unblocks (Phase B)

The 693-entry distillation map is also the **training-data shape for Phase
B**. Each entry pairs a noun with (a) a canonical colour family label and
(b) a canonical hex. That's exactly what a sentence-transformer fine-tune
needs to learn the noun → colour mapping in embedding space: positive pairs
are `(noun, family-name-prompt)` and `(noun, hex-derived-text)`; negatives
are sampled from non-matching families. The dataset is small enough to
hand-curate to high quality (the bias that fine-tuning needs), and broad
enough across materials / animals / plants / foods / natural phenomena /
household objects to span the space.

When Phase B ships, the runtime can keep the distillation lookup as the
"high-confidence shortcut" and route OOV terms through the fine-tuned
embedder — or drop the lookup entirely if the embedder generalises well
enough. Either way, the dataset is the load-bearing artefact.
