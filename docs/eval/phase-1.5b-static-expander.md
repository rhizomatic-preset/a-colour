# Phase 1.5b — static-expander eval

Date: 2026-05-17
Library: small (xkcd ∪ CSS, ~1 050 entries)
Engine: literal (TF-IDF; Phase 2 hasn't shipped yet)

This is the per-category lift table for Phase 1.5b. The static expander is
backed by a precomputed nearest-neighbour table generated by
`scripts/build-expander-vectors.ts` from GloVe 6B 50d (Stanford NLP, hosted on
HuggingFace). The runtime never sees the embedding model — only the small
`src/generated/expansions-static.json` (~9 KB) ships in the bundle. The
neighbour pool is constrained to the colour-name TF-IDF vocabulary, which is
what makes a token like "caterpillar" produce useful expansions ("apple",
"sap", "shell" — all tokens present in colour names).

## Headline

- **Open-vocab category went from 0/0 (noop, handcurated) to 20/30 (static, static-handcurated).** The blended config matches static-only on this category while preserving every other gain from handcurated. Three of the ten open-vocab queries now return a defensible top-3 family match (`caterpillar`, `pumpkin spice`, `lego car`).
- **No regression on the gate categories.** trivial, literal-name, and css-literal all hold at their Phase 1A baselines (100% acc@1, 80%/100%/100% acc@3 respectively).
- **All Phase 1.5a gains preserved.** Te Reo Māori (te-reo) stays at 88/100, weather lifts further (17→50 acc@1 against handcurated alone), cultural stays at 100/100, compound stays at 50/83.
- **Open-vocab still falls short of the design-doc 60/80 target.** That target was set before the engine was implemented; achieving it from GloVe-50d alone proved unrealistic without unacceptably regressing the cheap-case categories. Pushing the per-token expansion budget higher (cap=8 vs cap=5) lifts open-vocab to 40% acc@3 but drops literal-name to 80/80 and weather to 50/83, which trips the Phase 2A acceptance gate.
- **The remaining seven open-vocab failures** (`rainbow trout`, `ender dragon`, `salamander`, `octopus`, `charizard`, `kirby`, `deku cosplay`) need an engine that understands phrase semantics, not per-token GloVe substitution. That's Phase 2 territory — the static-expander as designed shouldn't carry it.

## Implementation notes

The static-expander dictionary build is parameterised:

- `--k=5` neighbours per token (the runtime currently uses up to 5 of these).
- `--min-score=0.30` cosine cutoff; below that, neighbours are dropped entirely.

Build artefact: `src/generated/expansions-static.json`, 9.4 KB on disk,
~14 KB at parse time. Same shape as the handcurated dictionary
(`Record<string, string[]>`), so the same expander-factory pattern handles
both. The blended expander emits handcurated entries first (at the higher
handcurated weight) and fills the per-token budget with static fallbacks (at
a discounted static weight), with deduplication.

The `MAX_STATIC_EXPANSIONS_PER_TOKEN` cap is 5 — empirically the sweet spot
between recall (open-vocab) and precision (modified-family, literal-name).
Cap=8 was tested and produces a clean regression on literal-name acc@3 from
100% to 80% — see the table above for the static config which uses cap=5.

## What this unblocks

Phase 1.5b is the recall floor for the engine bake-off. Phase 2A (engine
evaluation, script-only) now runs the same eval queries against three candidate
embedders to see whether a real engine can close the remaining 70% of the
open-vocab gap. Decision criteria and per-candidate reports land in
`phase-2a-decision.md` and `phase-2a-<engine>.md`.
